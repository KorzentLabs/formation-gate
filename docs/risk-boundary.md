# Risk Boundary

Formation Gate is intentionally limited to diagnostic reporting to prevent unintended side effects.

## What Formation Gate Does NOT Do

### No Source File Modifications
- Formation Gate never modifies source files
- No auto-fix or auto-formatting
- No code generation

### No Automatic Actions
- No automatic merging of PRs
- No automatic deployment
- No automatic publishing
- No automatic approval labels

### No Guarantees
- Does not guarantee correctness
- Does not guarantee security
- Does not guarantee compliance
- Does not guarantee production readiness
- Does not guarantee bug-free code

### No Network Calls
- No external network calls beyond GitHub API
- No secret fetching
- No third-party service integration

### No Secrets Required
- Does not require API keys
- Does not require tokens
- Does not require credentials

## Proof-Only Enforcement

All receipts generated by Formation Gate are marked with:
- `proofOnly: true`
- `executionAuthority: false`

This ensures:
- Receipts are diagnostic only
- No execution authority is granted
- No automatic actions can be taken based on receipts

## Fail-Closed Behavior

Formation Gate uses fail-closed behavior:
- Missing required fields cause failures
- Invalid configurations are rejected
- Errors are surfaced rather than hidden

## Deterministic Where Possible

Formation Gate aims for deterministic behavior:
- Same inputs produce same outputs
- Hash generation is deterministic
- Check results are reproducible

## Scope Limitations

Formation Gate only checks:
- Evidence presence
- Language patterns
- Receipt structure
- Configuration validity

It does not check:
- Code correctness
- Security vulnerabilities
- Performance characteristics
- Business logic validity
